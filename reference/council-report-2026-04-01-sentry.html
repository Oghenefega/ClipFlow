Sentry Error Tracking Integration for ClipFlow — April 1, 2026

How should ClipFlow integrate Sentry error tracking using @sentry/electron, and what should the relationship be between Sentry and the existing electron-log logger? Should all error tracking move to Sentry, or keep dual systems?

Chairman's Verdict

Where the Council Agrees

  • Manual process.on handlers are wrong. @sentry/electron hooks uncaughtException and unhandledRejection automatically. Adding manual handlers creates double-reporting or swallows errors. Remove them from the plan entirely.
  • Renderer Sentry.init() goes in src/index.js, before ReactDOM.render(). Not in App.js. No dissent.
  • electron-log stays, but its role changes. It remains the local diagnostic log. It does not compete with Sentry for crash handling. Two systems, two purposes: local file debugging vs. remote crash intelligence.
  • CRA source maps require tooling. Need craco or react-app-rewired + @sentry/webpack-plugin. Without this, production stack traces are useless.
  • The Contrarian's analysis was unanimously rated strongest by all 5 peer reviewers. The triple-handler collision insight is the critical finding.

Where the Council Clashes

Scope of Integration

First Principles + Executor: Keep it minimal — 4 touch points, iterate later. You're pre-launch with one tester.

Expansionist: Instrument everything now — perf monitoring, release health, Sentry.setUser().

Resolution: Foundation first. Expansion is valid but premature before the error pipeline works correctly.

startCatching() Resolution

Contrarian: Must remove startCatching() — it's a competing crash handler.

First Principles + Executor: Leave electron-log alone.

Resolution: Contrarian is right. startCatching() installs its own process.on('uncaughtException'). Remove it. Keep electron-log for routine logging.

Blind Spots the Council Caught

These emerged only through peer review — no individual advisor addressed them:

  • Offline error queuing. ClipFlow is a fully local pipeline. Users run long FFmpeg transcodes and gaming sessions without internet. Sentry must queue events and flush when connectivity returns.
  • GDPR/privacy consent. Sentry captures file paths, system info, potentially usernames. A commercial product needs a user-facing opt-in/out toggle for telemetry.
  • IPC boundary error correlation. Errors crossing the preload bridge appear as disconnected events. A shared session/trace ID is needed to link main ↔ renderer errors.
  • Electron 28 compatibility. @sentry/electron v4+ needs verification against Electron 28. A 5-minute check that prevents hours of debugging.

The Recommendation

Implement Sentry in four touch points, resolve the handler collision, defer everything else.

  1. src/main/main.js — Sentry.init({ dsn }) at the very top. No manual process.on handlers. Remove errorHandler.startCatching() from electron-log. Keep all log.info/warn/error calls.
  2. src/index.js — Sentry.init({ dsn }) before ReactDOM.render().
  3. App.js — Wrap top-level with Sentry.ErrorBoundary. Keep EditorErrorBoundary for recovery UI, add Sentry.captureException() inside its componentDidCatch.
  4. Source maps — Add craco + @sentry/webpack-plugin for readable production traces.

What you do NOT do yet: performance monitoring, transaction instrumentation, breadcrumb customization, user identification, release health dashboards.

Backlog items: GDPR opt-in/out toggle, offline queuing verification, IPC error correlation via shared session ID.

The One Thing to Do First: Verify @sentry/electron v4+ compatibility with Electron 28. Run npm install @sentry/electron, check for peer dependency warnings, confirm the SDK initializes in both processes. If it doesn't work, everything else is moot. This is a 10-minute task that gates the entire plan.

Advisor Responses

The Contrarian — ★ Unanimously rated strongest

The plan has three problems that will bite you in production.

1. Manual process.on handlers are redundant and dangerous. @sentry/electron already hooks uncaughtException and unhandledRejection automatically. Adding your own handlers will either double-report every crash or swallow exceptions before Sentry's handler sees them. Delete step 3 entirely.

2. "Keep electron-log completely untouched" is the wrong call. You have errorHandler.startCatching() which also intercepts uncaught exceptions. That's a third handler competing with Sentry's. You'll get race conditions. You need to either remove startCatching() or wire it as a Sentry beforeSend integration.

3. CRA + @sentry/electron renderer setup is fragile. CRA's webpack config is locked down. You can't easily add Sentry's webpack plugin for source maps without ejecting or using craco/react-app-rewired. Without source maps, every renderer error shows minified garbage.

Also missing: user consent for telemetry, rate limiting, PII scrubbing config, Electron 28 compatibility verification.

The First Principles Thinker

The real question isn't "how do I integrate Sentry" — it's "what failure information do I actually need to ship a commercial product?"

You're pre-launch with one tester. Sentry costs money at scale. Before wiring anything up, ask: what decisions will Sentry data enable that local logs won't? Right now, the answer is nothing.

Minimal correct architecture: Sentry.init() in main.js (SDK handles both processes), one AppErrorBoundary, keep electron-log for local diagnostics, add Sentry.captureException to EditorErrorBoundary's componentDidCatch. Four touch points. No manual process handlers. No architectural astronautics.

The dual system question is a false choice. electron-log is your local debug tool. Sentry is your remote crash intelligence. They serve different purposes. Don't merge them.

The Expansionist

Sentry isn't just error tracking. It's your product intelligence layer for a commercial app you haven't launched yet.

Instrument FFmpeg/Whisper pipelines with Sentry transactions for performance monitoring. Use release health (crash-free session rates) as your objective "ready to ship" metric. Breadcrumbs auto-capture console logs alongside errors — so electron-log and Sentry are complementary, not competing.

Renderer init goes in src/index.js before ReactDOM.render. Add Sentry.setUser() when the license system is wired — correlating errors to subscription tiers is critical business data.

Don't think of this as "adding error tracking." Think of it as instrumenting your product for commercial operation.

The Outsider

You're building two independent error systems and calling it a plan. Why keep electron-log "completely untouched" when it has session tracking, redaction, and structured reports? Six months from now, someone debugging a crash will check Sentry, see a stack trace, then hunt through local log files for context. That's not integration, that's accumulation.

The real question you're not asking: What happens when a paying user hits a crash and you need to correlate Sentry events with their local logs? You need a shared session/trace ID that both systems know about. Define that contract before writing code.

Also: CRA's production builds mangle React component names. Source maps are mandatory, not optional.

The Executor

Here's what you do Monday morning:

Step 1: Install @sentry/electron. That's it for dependencies.

Step 2: Two Sentry.init() calls. One in src/main/main.js at the very top. One in src/index.js before ReactDOM.render. Not in App.js — that's too late.

Step 3: Wrap the app root in Sentry.ErrorBoundary from the SDK. Keep EditorErrorBoundary for recovery UI — different jobs.

Step 4: Leave electron-log alone. They serve different purposes.

CRA needs @sentry/webpack-plugin via craco for readable stack traces. Check @sentry/electron v4+ Electron 28 compatibility. Ship it, iterate.
