## Key Decisions
- **A user cancel is NEVER a failed render.** It resolves `{canceled:true}`, the clip keeps/returns to its prior status, and NO partial files survive ‚Äî neither the temp overlay PNG folder (`cleanupOverlayFrames`) nor a half-written output `.mp4` (`unlinkSync`). Thumbnail is only made on success, so a cancel never produces one. (Fega explicitly asked "are half-baked files left on disk?" ‚Äî answer is no, both phases clean up.)
- **Two cancelable phases, one button.** 34% = the offscreen overlay-frame pass, BEFORE FFmpeg exists. "Kill ffmpeg" alone would not cancel an early render. `cancelActiveRender()` halts whichever phase is live (flag for the overlay loop, `proc.kill("SIGTERM")` for FFmpeg) plus a boundary race-guard.
- **NO installer this session (batching rule).** #140 is change ~#1 toward the next ~10-change batch. Source pushed; it rides the next batched installer OR Fega asks sooner. He can smoke-test it now via `npm start` if he wants ‚Äî the cancel button is live from source.
- **Partial-`.mp4` delete is defensive (try/catch).** It runs in the FFmpeg `close` handler, i.e. AFTER the process has exited and released the file handle, so the unlink succeeds. The try/catch only guards the astronomically-unlikely case Windows hasn't released the lock yet ‚Äî it would skip rather than crash, leaving the file. If Fega ever spots a stray partial after a LATE cancel, add a short retry. Not expected.
