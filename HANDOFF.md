## Watch Out For
- **DON'T cut an installer per fix.** Keep fixing + pushing source; build only at ~10 batched changes or on explicit ask. `.5` was the last build; everything since (#140, #141, #137, #138, #99) is source-only.
- **`data/clipflow.db` + `data/game_profiles.json` are always dirty (runtime churn) — never commit them.** Stage files explicitly; never `git add -A`/`.`. `tasks/mocks/` is untracked scratch — leave it.
- **#99 — style baseline writes are DIRECT `set()`, never setters.** Setters push undo entries; `setSegmentMode` triggers a re-chunk. The factory defaults MUST stay value-identical to the initial state (verified this session). If you add a new style field, add it to BOTH the factory AND the save/restore/template paths or it'll bleed again.
- **#138 — the `words[]` re-sync only fires when token count matches.** A text edit that changes word count (add/remove a word) keeps the old timed words (by design — timing wins; #89 re-chunk reconciles spelling on the next mode switch). Don't "fix" the mismatch branch to force a re-split — that's #95/#136 territory, deliberately left alone.
- **#141 — the peaks array can now be large (up to 100000).** Any new code consuming `waveformPeaks` must NOT spread it as function args (`Math.max(...peaks)`, `fn(...peaks)`) — use a loop. The cache JSON is now ~360KB for a 30-min source (was ~40KB); fine, but don't log the whole array.
- **#137 — only the SUBTITLE split was mapped.** Caption (`splitCaptionAtPlayhead`) and audio (`splitAtTimeline`) splits already operate in timeline space and were correctly left untouched. Don't "consistency-fix" them to use `toSource`.
- **Main-process files are NOT bundled by Vite** — the `main.js` waveform change was verified with `node --check`, not the renderer build. `npm run build:renderer` only covers renderer source.

## Logs / Debugging
- **Waveform** (`src/main/main.js:765` `waveform:extractCached`): logs `[waveform] extracting peakCount=N track=T`. Extraction is FFmpeg `-ar 1000` mono PCM → bucketed to peakCount peaks spanning [0, decoded audio duration]. Cache: `<projectsRoot>/<projectId>/.waveforms/<base>.<mtime>.<size>.<peakCount>.json` — peakCount in the key means a density change auto-invalidates old caches. A blob/featureless waveform on a long source = density too low (was the #141 symptom).
- **#99 style-bleed trace order:** clip open → `useEditorStore.initFromContext` → `subtitleStore.clearAll()` + `captionStore.initFromClip(null)` (baseline reset, SYNC) → async `storeGet` → `applyMergedTemplate(tpl)` → `restoreSavedStyles()` (per-clip saved style wins). If a style still bleeds, confirm the field is in the factory AND `restoreSavedStyle`'s mapping.
- **Reading the prod log on Windows:** `%APPDATA%\clipflow\logs\app.log`. From Bash: `powershell -NoProfile -Command '$p = Join-Path $env:APPDATA "clipflow\logs\app.log"; Get-Content $p -Tail 40'`.
- **Live settings (templates, defaults):** `%APPDATA%\clipflow\clipflow-settings.json` (electron-store, name `clipflow-settings`). `layoutTemplates` array, `defaultTemplateId`, `builtInTemplateDeleted`. Audited this session to confirm #99 — read it with `node -e` if you need to check a real template's completeness.
- **Build/verify:** `npm run build:renderer` (Vite, renderer only — DEFAULT after a renderer fix). Main-process files verify with `node --check <file>`. `npm start` smoke-launches prod-from-source; `exit code 143` after a `timeout`-killed launch is SIGTERM teardown, not a crash. The "renderer process gone: crashed (143)" + DawnCache/disk-cache "Access is denied" lines in the smoke test are the **already-running installed app holding the userData profile** — not a regression. **No test runner** — verify pure functions with `node -e`.
