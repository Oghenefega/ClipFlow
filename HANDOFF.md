## What Was Just Built (this session)
1. **Ripple-delete subtitle scramble fix** (`1f4018c`) — Ripple-deleting a clip on the timeline (the `track === "audio"` branch of `handleDelete`, `TimelinePanelNew.js:613`) closed the video gap correctly but mashed the subtitles. Subtitles live in **source time** and are *projected* onto the timeline through the NLE cut list (`visibleSubtitleSegments`, `timeMapping.js:220`), which already drops a deleted region's subs (`:249`) and re-times the rest — **the exact path trims use** (`trimNleSegmentLeft` touches only nleSegments, no subtitle mutation). `handleDelete` was ALSO manually ripple-shifting the subtitle store's source times on top of that projection (`subStore.rippleDeleteSegment`, `useSubtitleStore.js:927`), double-shifting by the sum of the deleted subs' own durations (not the clip's duration) → corruption. Fix: the clip-delete branch now just calls `deleteNleSegment(segId)` and lets the projection re-time. Removed the now-unused `nleSegments` dep from the callback. Side benefit: undo is fixed too (old path corrupted subs when undone).
2. **Mid-clip playhead-reset-on-play fix** (`1f4018c`) — Pressing play with the playhead parked mid-clip snapped it to the start. Root cause: the preview's rAF loop treats `<video>.currentTime` as the **source of truth** and overwrites the store playhead from it on the first frame of play (`PreviewPanelNew.js:888`); nothing ever pushed the playhead's position *into* the video except an explicit `seekTo`, so any video↔playhead drift (e.g. `onLoadedMetadata` parking the element at the first segment, `PreviewPanelNew.js:932`, without syncing the store) yanked the playhead back on play. Two-part fix: (a) `togglePlay` now re-seeks the video to `currentTime` on play start (`usePlaybackStore.js:38`), and (b) the rAF loop ignores frames while `video.seeking` (`PreviewPanelNew.js:880`) so a just-issued seek can't be clobbered by a stale position. Kept the existing "replay from start when truly at the end" behavior.
3. **Cut 0.1.8-alpha.11** (`60177bf`) — bump + CHANGELOG + full `npm run build`. Carries both fixes above PLUS everything still riding from alpha.9/alpha.10 (see Next Steps).
4. **Lesson distilled** — "don't present an UNREPRODUCED playback/state diagnosis as fact — reproduce or tag UNCONFIRMED + fix the class; surface ambiguous post-edit numbers instead of picking the confirming reading" → `clipflow-trace-verify` (Distilled Lessons). ## Key Decisions
- **Bug 1 fix is "stop double-handling," not "fix the math."** The subtitle re-timing belongs to ONE layer (the source→timeline projection). The clip-delete handler was a second, conflicting writer. Removing it (rather than making its shift-amount correct) is right because the projection is already the single source of truth for every other consumer (preview, transcript, render) and matches how trims behave. Verified `trimNleSegmentLeft` does zero subtitle mutation and trims work correctly — that's the proof the projection alone is sufficient.
- **Bug 2 fixed as a CLASS, not one trigger.** Clean-state traces mapped mid-clip play *correctly*, so I could not reproduce the exact desync statically. Rather than guess one trigger, enforced the invariant "play always starts from the visible playhead" (re-seek on play + ignore in-flight-seek frames), which fixes the desync however it arose. Got Fega's approval before touching playback core, since it's central.
- **Left the dead `rippleDeleteAudioSegment` (`useEditorStore.js:497`) untouched** — zero callers (confirmed by grep); it's the legacy audioSegments model. Not my bug, didn't touch it (surgical-changes rule).
- **No GitHub issues filed** — both were inline fixes reported + fixed in the same session/commit (per `.claude/docs/issue-filing.md`, don't file for inline same-commit fixes). CHANGELOG + commits are the paper trail.

## Watch Out For
- **Install alpha.11, not 10/9.1.** The in-app notifier picks newest by mtime → alpha.11 surfaces correctly. Settings → bottom must read **v0.1.8-alpha.11** to confirm the install took.
- **The `<video>` element is the source of truth for the playhead; the store only mirrors it.** Any code that moves the video (a new seek path, a reload, a metadata handler) MUST also keep the store `currentTime` in sync, or the next play will yank the playhead to the video's real position. `onLoadedMetadata` (`PreviewPanelNew.js:932`) still seeks the video without syncing the store — it's only safe today because it fires on clip-open (store already 0) and `togglePlay` now re-asserts on play.
- **Raw `editSegments` is source-wide — do NOT read it as the clip's content, and do NOT mutate it in response to an NLE (video) edit.** The projection (`getTimelineMappedSegments`/`visibleSubtitleSegments`) handles clip-window clipping AND re-timing on cut-list changes. Mutating subtitle source-times to "follow" a video edit double-applies the shift (this session's bug 1). Direct subtitle-track edits (`track === "sub"`) DO legitimately mutate editSegments — that's editing the subtitle data itself.
- **The 3 shared utils are still CJS** (`resolveSubtitles`, `cleanWordTimestamps`, `wordRepair`) — don't re-add ESM `export`/`import`; the main-process `require()` in render.js depends on it.
- **`package.json` silent-strip gotcha** ([[project_package_json_strip]]) — if builds break for no reason, check it has its `scripts`/`build`/`devDependencies` blocks and `git checkout HEAD -- package.json`.
- **`data/clipflow.db` + `data/game_profiles.json` always dirty — never commit.** Stage files explicitly; never `git add -A`. `tasks/mocks/` is untracked scratch.
