LLM Council Report — Preview Subtitle Rendering

Council session — March 31, 2026

Question: How should ClipFlow's project preview tab render subtitle/caption overlays to match the editor's template from the moment clips load — without requiring users to open each clip in the editor first?
CLEAR PATH FORWARD Extract a shared pure function. One new utility file, ~50 lines. No architectural redesign needed.

Advisor Positions

Contrarian Questions if preview even needs subtitles
First Principles One shared pure function layer
Expansionist Canonical renderer for everything
Outsider Stop building two renderers
Executor 50 lines, 2-3 hours, ship it

● Aligned   ● Conditional

Chairman's Verdict

Where the Council Agrees

Unanimous

One shared pure function. All 5 advisors independently converged on: extract a pure function that takes words + template and returns display-ready segments. No React, no Zustand. Editor and preview both consume it.

Unanimous

Skip karaoke/animation in preview. At 220px, word-level highlighting and scale animations are wasted. The preview needs: correct text (punctuation stripped), correct position (yPercent), correct style (font/color/effects).

Strong Agreement

Default template as fallback. Unsaved clips use the user's default template (from electron-store). Saved clips use their persisted subtitleStyle. The fallback chain is: clip.subtitleStyle ?? defaultTemplate.

Strong Agreement

"Two renderers" is the root cause. The preview diverged because it was built as a separate, simplified renderer. Every editor feature addition widens the gap. Fix the architecture, not the symptoms.

Where the Council Clashes

Scope Disagreement

Preview-only utility vs. canonical renderer. The Executor says build a focused buildPreviewSubtitles.js (~50 lines, ships fast). The Expansionist says build a canonical renderer the editor also uses. Chairman sides with: build the preview utility now. The editor's rendering pipeline is deeply integrated with Zustand stores and karaoke state — refactoring it to use a shared function is a separate, larger task that risks regressions.

Validity Questioned

Does preview need subtitles at all? The Contrarian asks why render subtitles at 220px. Chairman overrules: yes, it does. The user explicitly wants to preview clips with subtitles in the project tab. This is a confirmed product requirement, not an assumption.

Blind Spots the Council Caught

Critical — Verified Resolved

Word data availability. 3 of 5 peer reviewers flagged: does clip.words actually exist on clips that have never been opened in the editor? Answer: Yes. Word timestamps live at clip.subtitles.sub1[].words[] and are written at pipeline time (ai-pipeline.js line 601). The data is always there for pipeline-generated clips.

Important

punctuationRemove not persisted to clip. Even after editor save, the clip object gets subtitleStyle (fonts, colors, position) but NOT the punctuationRemove config. The preview must read punctuation settings from the template, not the clip. Fix: persist punctuationRemove as part of subtitleStyle on save, OR always read it from the active template.

Edge Case

Template change after clip save. If user saves clips with Template A, then changes default to Template B, what should unsaved clips show? B (default). What should saved clips show? Their saved style. This is already handled by the clip.subtitleStyle ?? defaultTemplate fallback.

The Recommendation

Build buildPreviewSubtitles.js — a pure function that takes (subtitleSegments, template) and returns display-ready subtitle data. It does 3 things:

  • 1. Gather words from clip.subtitles.sub1 segments, run through segmentWords()
  • 2. Strip punctuation per template.punctuationRemove config (~15 lines, copy from editor's stripPunct)
  • 3. Return segments + style object built from template fields (font, color, yPercent, stroke, glow, shadow, bg)

In ClipVideoPlayer, replace the inline buildSubPreviewStyle() + manual segmentation with a single useMemo call to this function. Template resolution: clip.subtitleStyle ?? previewTemplate.

Also persist punctuationRemove to clip.subtitleStyle during editor save so saved clips render correctly regardless of template changes.

The One Thing to Do First

Create buildPreviewSubtitles.js with the pure function, wire it into ClipVideoPlayer, and persist punctuationRemove on editor save. This is a single focused task that fixes all three symptoms: wrong positioning, missing punctuation stripping, and template mismatch.

Advisor Responses

Contrarian "Why does preview need subtitles at 220px?"

Raised three flaws: (1) hidden state problem when default template changes after clips are saved, (2) punctuationRemove not persisted means saved clips still render wrong, (3) performance cliff with 10-20 clips running segmentation. Questioned whether preview subtitles are even needed.

Chairman's note: State problem is handled by the fallback chain. Performance is fine with useMemo (runs once per clip, not per frame). Preview subtitles are a confirmed user requirement.

First Principles "Rendering logic exists in two places — that's the bug"

Reframed the problem: this isn't a preview bug, it's the tax of duplicated rendering logic. Proposed a single pure function renderSubtitleSegments(words, template) called by both editor and preview. The absence of saved style is the default state — design for it explicitly.

Chairman's note: Correct diagnosis. The pure function approach is the consensus path forward. Editor integration is a future improvement.

Expansionist "Build the canonical subtitle renderer"

Saw the opportunity: a template-aware render function decoupled from the editor can be used in render queue preview, bulk style workflow, and thumbnail generation. "Don't build a preview approximation. Build the canonical subtitle renderer."

Chairman's note: Right long-term vision. For now, the preview utility is step one. It naturally evolves into the canonical renderer as other consumers adopt it.

Outsider "Opening editor to fix preview is a data leak"

Identified the UX absurdity: a new user would find it baffling that opening a clip in the editor "fixes" the preview. That's not a feature, it's a broken data flow. The preview should never depend on whether someone has visited a different view.

Chairman's note: This is the clearest articulation of why this matters for a commercial product. Users will report this as a bug.

Executor "50 lines, 2-3 hours, ship it"

Gave the implementation path: create buildPreviewSubtitles.js with three steps (segmentWords, stripPunct, return {segments, style}). useMemo keyed on clip.words runs once per clip. Skip karaoke/animation. Per-clip fallback: clip.subtitleStyle ?? previewTemplate.

Chairman's note: This is the implementation plan. Adopted as-is with the addition of persisting punctuationRemove on save.
