Council Report: Project Folders Feature

ClipFlow Projects View — April 1, 2026

Council Verdict

Build flat, user-created folders stored as metadata in electron-store. Sidebar folder list with counts, multi-select with floating action bar, right-click context menus. Color dots, no icons. Delete folder returns projects to "All Projects" — never deletes files. Skip nested folders, drag-and-drop, smart folders, tags, and templates for V1.

Where the Council Agrees

Flat folders, not nested. Every advisor who engaged with folders said one level deep. No one argued for hierarchy.
Metadata layer, not filesystem. Universal: folder membership stored in electron-store, not by moving project files into OS directories. Projects stay where they are.
The Executor's data model is correct. All 5 peer reviews named it strongest. { id, name, color, projectIds[] }
Skip drag-and-drop for V1. Multi-select + action bar achieves the same result without DnD complexity.
Delete folder must NOT delete projects. Deleting a folder removes the grouping. Projects return to All Projects. File deletion is separate.

Where the Council Clashes

Folders vs. Tags vs. Filters
First Principles + Outsider argue the real need is filtering, not folders — building folders may paint you into a corner when you need tags later. Executor + Contrarian say: user asked for folders, folders are the simplest mental model, ship folders.
Resolution: Folders-as-metadata (projectIds arrays) is structurally identical to single-value tags. Extending to multi-tag later is straightforward. Not a dead end.
How minimal is too minimal?
Contrarian: hardcoded Dev/Prod toggle. First Principles: single archive boolean. Executor: user-created folders with colors.
Resolution: ClipFlow is a commercial product heading to release. Hardcoded categories won't survive real users with their own mental models. Build user-created folders.

Blind Spots Caught in Peer Review

Migration path: Existing projects need to appear in "All Projects" on upgrade. Must be explicit — zero-friction, no prompts.
Folder + pipeline filter interaction: Folder selection must compose WITH existing status filtering (ready/processing/done), not replace it. Folder narrows, then status narrows further.
Sort order within folders: Use ordered projectIds[] array. Default to existing sort (status then date). Manual reorder is V2.

The Recommendation

Data model in electron-store:

folders: [{ id, name, color, projectIds: [] }]

UI changes:

  • Sidebar folder list (or top bar tabs) with project counts
  • "All Projects" pseudo-entry always at top, showing everything
  • Projects not in any folder appear under "All Projects" (no "Uncategorized" noise)
  • Multi-select via checkboxes, floating action bar: "Move to folder" dropdown
  • Right-click on projects: "Move to folder" submenu
  • Right-click on folders: Rename, Change color, Delete
  • "+ New Folder" button in sidebar
  • Color dot only — no icons, no emoji

Critical behaviors:

  • Existing filters (status, pipeline) compose with folder selection
  • Delete folder prompt: "This will remove the folder. Your X projects will still be available in All Projects."
  • On upgrade: all existing projects in "All Projects", no folders created — zero friction
  • projectIds array is ordered for future manual sort

Skip for V1: nested folders, drag-and-drop, smart folders, tags, templates, batch render/schedule, cross-folder references

The One Thing to Do First

Define the electron-store schema for folders and write the IPC handlers (create, rename, delete, recolor, addProjects, removeProjects) in main.js — then expose through the preload bridge. Data layer first, UI second.

Advisor Responses

The Contrarian ▼

The fatal flaw is scope creep disguised as a simple feature. Nested folders, drag-and-drop, smart folders — all traps.

Store folder assignments in electron-store with a reconciliation check on startup. Multi-select + "Move to folder" dropdown does drag-and-drop's job in 10% of the dev time.

What's actually needed: two hardcoded categories (Development/Production) with a toggle. Solve the real pain point in an afternoon. If that proves insufficient after two weeks of real use, then build the full folder system with actual usage data.

Key insight: Don't build folder infrastructure for a one-person test phase. Build a filter.

The First Principles Thinker ▼

The real problem isn't folders — it's filtering. Folders force a single axis of organization. "Do I put this Valorant TikTok in the Valorant folder or the TikTok folder?"

Minimum viable solution: a single archive/hide boolean. Right-click, "Archive." Toggle "Show archived." Ships in an hour.

If building folders anyway: flat, metadata-only, one folder per project, invisible until first creation.

Key insight: Will this need tags/smart filters within 6 months? If yes, build the thin metadata layer that supports all three.

The Expansionist ▼

Smart folders that auto-populate from filters. Folder-level batch operations: render all, schedule all. Folder templates with pre-configured platform defaults. Color + emoji tags. Cross-folder project references.

Key insight: Model folders as lightweight metadata layers, not filesystem directories. This keeps every expansion path open.

Note: Peer review unanimously rated this the least grounded response — too much scope for current stage.

The Outsider ▼

You said "folders" but described "tags." Pick one. Don't restructure the filesystem.

V1: Uncategorized pseudo-folder, sidebar with counts, click to filter, right-click to move.

Key insight: Delete folder must distinguish "remove from folder" vs "delete project files from disk." These are extremely different operations and conflating them will cause data loss.

The Executor ▼

Flat folder list in electron-store: { id, name, color, projectIds[] }. Sidebar folder panel with "All Projects" at top, user folders below, "+ New Folder" at bottom. Click folder to filter. Multi-select via checkboxes, floating action bar with "Move to Folder" dropdown.

Right-click folder: Rename, Change Color, Delete. Delete shows confirmation with project count.

Skip: nested folders, drag-drop, auto-folders, folder icons. Color dot is enough. Six to eight files, one session.

Rated strongest by all 5 peer reviews.

Peer Review Highlights

Unanimous strongest: The Executor (Response E) — the only concrete, buildable spec.

Unanimous weakest: The Expansionist (Response C) — feature fantasy for a solo-tester product.

Recurring blind spots flagged:

  • Sort order within folders (flagged by 2/5 reviewers)
  • Folder + pipeline status filter interaction (flagged by 1 reviewer — but critical)
