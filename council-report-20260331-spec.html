Blind Spots Caught in Peer Review

Critical

Input contract is undefined. Does the function receive word objects with timestamps? What shape? What happens with malformed input, timestamp drift, non-monotonic timestamps, or overlapping tokens from Whisper? The entire spec assumes clean input. That assumption is wrong for a Whisper pipeline.

High

No conflict resolution table. When two rules fire on the same token ("um um um" = filler + repeat, "you know?" = filler + sentence ender), there is no declared winner. This is normal for gaming audio, not an edge case.

4 Fixes Before Implementation

1 Define the input contract. Function signature, word object shape (fields, types, units), what to do with malformed/overlapping/drifting timestamps (skip, clamp, throw).
2 Write a conflict resolution table. Every pair of rules that can fire on the same token — declare the winner. "you know?" → hard wall wins. "um um um" → repeat wins. 5-10 rows, not fifty.
3 Specify Rule 5 with worked examples. 3-word over limit: split 2+1 (first 2 + last 1). 2-word over limit: split 1+1. Single word over 20 chars: keep as-is (can't split a word). Show the math.
4 Resolve the 0.7s / 1.0s relationship. One sentence: "0.7s is a hard wall (pre-partition). 1.0s forward look only operates within partitions and is irrelevant when a hard wall already fired." Or whatever the answer is. State it.

The One Thing to Do First

Write the input contract. One function signature, one type definition, one paragraph on what "clean enough" Whisper output means. Everything else operates on the input — if the input schema is ambiguous, every rule is ambiguous.

Advisor Responses

Contrarian "Three gaps that will cause implementation bugs"

Gap 1: Filler "like" detection is handwaved — no way to distinguish filler from verb. "I like this game" vs "like, I dunno." The implementer will guess, it'll be wrong.

Gap 2: Repeated phrase detection needs a minimum recurrence count and adjacency requirement. A phrase at 0:05 and 2:30 shouldn't trigger the same as "go go go."

Gap 3: Priority conflicts unresolved — "um um um" hits both filler and repeat rules. Which wins? Also: char split direction not specified, single long words unaddressed.

First Principles "Missing layer: input normalization"

What counts as a "word" is undefined — hyphenated terms, contractions, "360fps." Filler detection needs a concrete heuristic, not "like when filler." The 6 test cases capture behavior not intent.

Missing layer: input tokenization/normalization before hard walls. Add a named pre-processing stage. Define "filler" with a concrete detection rule, even a naive one.

Expansionist "Correctness is the floor, not the ceiling"

Spec solves for correctness but not virality. Gaming subtitles need emotional pacing — high-energy words isolated for impact. Adjacent opportunities: emphasis detection via audio energy levels, style presets per mode ("Streamer" vs "Documentary").

Note: Rated weakest by all 5 peer reviews. Valid future direction but not a spec-level concern.

Outsider "Rule 5 will cause the next regression"

0.7s hard wall vs 1.0s forward look — what happens between? Rule 5 (char split) is the vaguest rule. "um I" creates two ugly tiny segments. Gap closing direction unspecified (earlier extends or later starts early?).

Test cases mentioned but not shown inline. Fix Rule 5 first — it has the highest regression probability.

Executor "Two blockers, then it's mechanical"

Blocker 1: "you know?" is both a filler phrase AND a sentence ender. Hard walls run first, so it never reaches filler isolation. Is that correct? Spell it out.

Blocker 2: 20-char split needs worked examples. Single-word partitions > 20 chars undefined.

Pre-partitioning is architecturally sound. Add edge case tests, then implementation is a mechanical translation job.

Peer Review Highlights

Strongest response: Outsider (D) — 3/5 votes. Catches highest-density implementation blockers.

Weakest response: Expansionist (C) — 5/5 votes. Feature pitch, not spec critique.

Critical blind spot (all 5 reviews): Input contract is undefined. The function's input schema — word object shape, timestamp reliability, malformed input handling — is the foundation everything sits on. Without it, every rule is undefined behavior.

Second blind spot: No conflict resolution table. When rules collide (normal for gaming audio), no declared winner.
