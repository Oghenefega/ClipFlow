LLM Council Report

Subtitle Segmentation Approach Review — March 31, 2026

Question: Is ClipFlow's subtitle segmentation approach fundamentally sound? The algorithm is a ~250-line procedural chunker with 6+ priority-ordered rules inside a Zustand store. It has regressed 3 times on the same 3 bugs. This is ship-blocking.

Advisor Positions

Contrarian Extract + Test
First Principles Extract + Test
Expansionist Ship, Fortify Later
Outsider FSM Rewrite
Executor Pre-partition + Ship

● Extract first   ● Test first   ● Ship now   ● Rewrite

Peer review consensus: 5/5 reviewers picked First Principles (extract + test) as strongest response. 4/5 flagged Expansionist's "ship now" as biggest blind spot.

Blind Spots Caught in Peer Review

Blind Spot

Whisper output is noisy. The algorithm receives missing punctuation, merged words, and mis-timed boundaries. Tests built against clean prose test the wrong thing. The suite must use actual Whisper transcripts from gaming clips.

Blind Spot

No written spec for "correct" segmentation. Without a canonical definition, tests only codify current behavior — a wrong algorithm with tests is still wrong.

Blind Spot

Bad subtitles reach the creator's audience. The commercial risk is asymmetric — broken subtitles ship to viewers, not just the creator's private workflow.

The Recommendation

The approach is fundamentally sound but architecturally misplaced and operationally unprotected. The algorithm's complexity is justified by the content format. The regression problem is not that the rules are wrong — it is that there is no floor under them.

Do not rewrite to FSM. Do not simplify the rules. Do not ship without a safety net.

The correct path: extract the algorithm to a pure function, write a segmentation spec (one page, explicit rules and thresholds), encode the three known regressions as named test cases using actual Whisper output, then run those tests before every future change. The Executor's pre-partition idea (sentence/pause boundaries as hard walls before chunking) is the best structural insight — implement it after extraction and testing, not instead of them.

The One Thing to Do First

Write the segmentation spec before touching any code. One page. Define what a correct subtitle segment is: maximum word count, sentence boundary characters, pause split threshold, what constitutes a repeated phrase, and the expected output for the three known regression cases. This spec becomes the definition of done for every future change.

Advisor Responses

Contrarian "The regression pattern is the real problem"

Three separate regressions of the same three bugs tells you there is no regression test suite. Every "fix" is a manual eyeball pass. Rules 0b/0c (stateful phrase recall) interact badly with Rule 1 (sentence boundaries) — a phrase protected earlier will be recalled later, but what happens when it falls after a sentence-ending period? The code says priority order, but stateful phrase memory doesn't reset at sentence boundaries. That's a latent bug.

The 250 lines inside a Zustand store action is untestable in isolation. You cannot write a unit test without spinning up React context. The actual ship risk is not launch day — it's breaking on a clip that gets traction, embarrassing the product.

First Principles "The real problem is architecture, not algorithm"

The actual question is "why does the same logic keep regressing?" — because it's a stateful blob inside a Zustand store action, designed for state transitions not parsing pipelines. The segmentation algorithm is a pure function: words-with-timestamps in, segments out. It should live in a standalone module with zero framework dependencies.

Once it's a pure function, write 20 hardcoded test cases covering every regression. The algorithm itself is probably fine — the complexity is warranted for gaming speech. But complexity without test coverage is a time bomb that's already detonated three times.

Expansionist "Gaming speech complexity is the differentiator"

Gaming speech isn't an edge case — it's the defining characteristic. The phrase detection, gap logic, sentence boundary rules are exactly what a gaming-specific product should be doing. Don't simplify — extract and test. This becomes a marketing story: "Built specifically for how gamers actually talk."

Ship what you have. Then fortify with unit tests covering known failure patterns. Every regression would have been caught before shipping. That investment pays compound returns.

Outsider "This is a finite state machine problem"

The three regressions are the same design failure in three costumes. You're solving a sequencing problem with a priority system that fails when two rules fire simultaneously. The rule list keeps growing because each edge case requires a higher-priority override. The next regression will require Rule 0d.

The segmentation problem is fundamentally a finite state machine problem. Deterministic state transitions eliminate rule priority issues. Pull into a pure function. Write three test cases per regression.

Executor "Pre-partition as hard walls, ship now"

Pre-partition the word list into sentence-boundary groups and pause-boundary groups as hard walls before any chunking runs. Chunking only operates within partitions. This is a 30-minute change: write partitionWords(words) that splits on .!? and gaps > 0.7s, then run the existing 3-word chunker on each group independently.

The reason it keeps regressing is rules run on a flat word list and compete. Partitioning first makes the rules physically unable to cross boundaries they shouldn't.

  • The three documented regressions in lessons.md ARE the test suite — just in the wrong format
  • No canonical spec for "correct" segmentation exists — tests without a spec just codify current bugs
  • Bad subtitles ship to the creator's audience, raising stakes beyond typical developer-facing bugs
