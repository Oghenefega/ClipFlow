Spec Review: Project Folders v1.0

Council reviewed the implementation spec — April 1, 2026

Spec Changes Required 7 FIXES

1. Define move-from-old-folder behavior When addProjects targets a project already in another folder, what happens? Silent move? Confirmation? The IPC handler must return { movedFrom: oldFolderName } so the UI can show a toast. This is the #1 ambiguity in the spec.
Flagged by: Contrarian
2. Add persistence failure handling Every mutating IPC handler needs try/catch returning { success: false, error }. UI must handle errors gracefully. One paragraph covers this.
Flagged by: Contrarian + First Principles
3. Add project deletion cleanup path When a project is deleted via existing project:delete, its ID must be pruned from folder projectIds. Specify who owns this: reconciliation on next folder:list, or explicit call during delete flow.
Flagged by: Contrarian + First Principles
4. Reduce IPC handlers from 8 to 6
  • Merge rename + recolor → updateFolder(id, patch)
  • Make addProjects(folderId: null) handle unassign (drop separate removeProjects)
  • Keep: list, create, updateFolder, delete, addProjects, reorder
Flagged by: Contrarian + First Principles
5. Specify empty folder state When a folder has zero projects, the main panel shows "No projects in this folder" message with a hint to use "Move to Folder" from All Projects.
Flagged by: Outsider
6. Specify "All Projects" interaction rules "All Projects" cannot be renamed, deleted, or recolored. Right-click does nothing (no context menu). Count = total projects, not just unassigned.
Flagged by: Outsider + Executor
7. Add undo toast for destructive folder operations Folder deletion and bulk move show a 5-second undo toast. "Folder deleted. [Undo]" / "3 projects moved to Production. [Undo]". This is data safety for a paid product, not scope creep.
Flagged by: All 5 peer reviews (unanimous blind spot)

Spec Is Fine As-Is NO CHANGE

One project per folder Council debated many-to-many but one-per-folder is correct for V1. Simpler mental model, matches physical folder metaphor. Tags are a different feature if needed later.
No drag-and-drop in V1 Right-click "Move to folder" is adequate for <100 projects. DnD is a V1.1 enhancement.
No keyboard shortcuts in V1 Folder operations are low-frequency. Shortcuts are polish, not structure.
160px sidebar width Fine for V1. Collapsible sidebar is V1.1.
Simple folder counts (no status breakdown) Single number, not "3 ready, 2 drafts." Status filtering within a folder handles this.
Migration path (existing projects start unassigned) Already addressed in spec. Correct and sufficient.

Open Questions for You DECIDE

1. Duplicate folder colors — allow or prevent? Two blue folders: confusing or non-issue? Show "already used" indicators on color picker, or ignore?
2. Folder ordering in sidebar — creation order or manual reorder? Creation order is simpler. Manual drag-reorder adds sortOrder field. Which for V1?
3. Folder limit? Cap at 20 to prevent sidebar scroll? Or unlimited?

The One Thing to Fix First

Define the move-from-old-folder behavior. When addProjects targets a project already in another folder: silent move with toast notification ("Moved from X to Y"), not a confirmation dialog. The IPC response includes movedFrom so the UI can display it. This is the single interaction that causes the most confusion if left ambiguous.
